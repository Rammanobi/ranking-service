Overview

A small backend + frontend that records user "transactions" (point-earning events), exposes a per-user summary, and ranks users on a leaderboard using more than one factor — built to demonstrate API design, data consistency, duplicate-request handling, concurrency safety, and abuse-resistant ranking.

How to run it locally

Backend

cd ranking-service/backend
python -m venv .venv
source .venv/Scripts/activate      # Windows Git Bash
# .venv\Scripts\Activate.ps1       # Windows PowerShell
# source .venv/bin/activate        # macOS / Linux

pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000

The API is now at http://localhost:8000 (interactive docs at /docs). A SQLite file backend/data.db is created automatically on first run.

Tests

cd ranking-service/backend
python -m pytest tests/ -v

Frontend

Plain static files, no build step. Open frontend/index.html directly, or serve it: 

cd ranking-service/frontend
python -m http.server 5500

Type the backend URL into the "API" field at the top of the app (defaults to http://localhost:8000) and click Save. A green dot means the API is reachable.

Live deployment

Backend → Render: push to GitHub, then in Render use New → Blueprint, pointing at this repo — it picks up backend/render.yaml automatically. Render's free disk is ephemeral, so attach a persistent disk (or swap SQLite for Postgres) if you need data to survive restarts.

Frontend → GitHub Pages / Netlify / Vercel: deploy the frontend/ folder as-is — it's static HTML/CSS/JS, zero config. CORS is wide open in main.py specifically so the frontend can live on a different domain than the API.

POST /transaction

Records a point-earning event for a user.

{ "user_id": "alice", "amount": 50, "idempotency_key": "alice-2026-06-23-001" }
  • user_id — 1–64 chars, letters/digits/-/_ only.
  • amount — number, 0.01 ≤ amount ≤ 100000. Rejected if missing, negative, zero, NaN/Infinity, or out of range.
  • idempotency_key — client-supplied string (1–128 chars) identifying this transaction attempt; reuse it on retries.

Responses: 201 first time seen · 200 with duplicate: true if the key was already processed · 422 on validation failure · 429 if rate-limited.

GET /summary/:userId

{
  "user_id": "alice",
  "total_points": 120.0,
  "transaction_count": 3,
  "active_days": 2,
  "ranking_points": 87.0,
  "rank": 4
}

total_points is the raw sum of all transactions. ranking_points is the score actually used for the leaderboard — not the same value by design (see below). Returns 404 if the user has no transactions yet.

GET /ranking

Leaderboard ordered by ranking_points descending (ties broken by total_points, then user_id). Supports ?limit= (default 100, max 500) and ?offset= for pagination.

[
  {"rank": 1, "user_id": "bob", "total_points": 300.0, "active_days": 1, "ranking_points": 213.0},
  {"rank": 2, "user_id": "alice", "total_points": 120.0, "active_days": 1, "ranking_points": 87.0}
]

How ranking is calculated

A pure sum of points is easy to game — one oversized transaction can take #1 with zero sustained engagement. So the score combines two factors:

ranking_points = 0.7 * capped_points + 0.3 * (active_days * 10)
  • capped_points — sum of the user's transaction amounts, but each individual transaction can contribute at most 5,000 points toward this sum (MAX_RANKING_CONTRIBUTION_PER_TRANSACTION in config.py). The raw, uncapped total is still shown as total_points.
  • active_days — number of distinct calendar days the user has at least one transaction on. Rewards consistent activity over a one-off spike.

Weights and the per-day bonus live in config.py as named constants — fully auditable and tunable, not a black box.

How duplicates and abuse are prevented

Duplicate processing (idempotency)

  • Every POST /transaction requires a caller-supplied idempotency_key.
  • transactions.idempotency_key has a UNIQUE constraint in SQLite.
  • On insert, the code checks for an existing row with that key first; if found, it returns the original transaction with duplicate: true instead of inserting again.
  • Race safety: the whole check-then-insert sequence is wrapped in a process-wide lock, and the database's own UNIQUE constraint is the fallback if two identical requests somehow race past that — caught and converted to the same idempotent response.

Consistent aggregate updates

A transaction insert and the corresponding summary/ranking updates happen inside one SQLite transaction — both land or neither does.

Abuse prevention beyond dedup

  • Rate limiting — max 10 transactions per user per 60-second sliding window, independent of idempotency keys.
  • Per-transaction ranking cap — stops one inflated transaction from buying the top rank.
  • Strict input validation — restricted character sets, bounded amounts, explicit NaN/Infinity rejection.

Assumptions & trade-offs

  • SQLite, single process — zero external setup, real ACID transactions. Multi-instance deployment would need the write lock and rate limiter moved to shared state (e.g. Redis).
  • No authentication — user_id is taken at face value from the request body; a real system would derive it from an authenticated session.
  • Rate limiter is in-memory — resets on restart, not shared across instances.
  • Transactions are always positive — no refund/penalty modeling.
  • Ranking weights are static constants, not learned/adaptive — simpler and fully auditable.
